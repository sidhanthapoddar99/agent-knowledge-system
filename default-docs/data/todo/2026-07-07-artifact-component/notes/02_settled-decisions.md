# Settled decisions — first-class artifacts (component + route)

This is the finalized decision record a downstream implementer cites. It pins
the choices the user has **settled** for the artifact content type and its
reserved route, with the rationale attached so the decisions survive a cold read.
The live deliberation — including the threads still open — is in
[../brainstorm/01_discuss_component-and-route-architecture.md](../brainstorm/01_discuss_component-and-route-architecture.md);
this note records only what is *decided*. The feature mirrors the first-class
diagram pipeline shipped in [[2026-04-10-editor-diagrams]].

## Scope of this record

This issue **plans**; it does not implement. The decisions below constrain the
implementation, but the actual code lands through the subtasks
(`10_component`, `20_route`, `30_reserved-url-guard`, and the skill/docs
workstreams). Where a decision leaves a sub-choice open, it is marked as such and
points at the brainstorm — an implementer must not read silence here as
permission to invent; they should resolve the named open thread in its subtask.

## Decided — the dedicated URL is the primitive

**Decided (sidhantha, 2026-07-07):** `/artifacts/<path-to-artifact>` is the
**primary** full-view mechanism, and the in-content embed is built *on top of*
it — not the other way around.

The reasoning is structural, not aesthetic:

- **The iframe needs a `src` regardless.** Embedding an artifact in the content
  area means pointing an iframe at a URL that serves the raw artifact HTML. That
  URL *is* the `/artifacts/<path>` route. So the route is not an optional
  companion to the embed — it is the foundation the embed cannot exist without.
  Building the route first is building the embed's own dependency.
- **Durable, bookmarkable, shareable.** A real URL can be linked from other docs,
  pasted into an issue, sent to a teammate, and reopened later. It survives
  outside the page that embeds it.
- **It sidesteps the Fullscreen API's weaknesses.** The browser Fullscreen API
  (the obvious alternative for a "big view") carries intrusive browser chrome,
  has awkward escape/exit behaviour, and — decisively — is not linkable. A URL has
  none of those problems.

Consequence for implementers: build `20_route` as real infrastructure, not as a
convenience wrapper. The embed in `10_component` sets its iframe `src` to this
route (with a `?v=<mtime>` cache-buster, per the diagram precedent at
`astro-doc-code/src/loaders/diagram-pages.ts:65-66`).

## Decided — embed affordances: open-full-page primary, expand secondary

**Decided (sidhantha, 2026-07-07):** the in-content embed carries an **"open full
page"** affordance as its primary control — a link to the `/artifacts/<path>`
route. A lightweight **in-place expand** (grow the iframe to fill the viewport
without navigating) **may** exist as a *secondary, optional* affordance.

Rationale: the full-page route already delivers the "big view" with a real,
shareable URL, so the open-full-page button is both trivial (an `<a href>`) and
complete. An in-place expand adds no new capability — only the convenience of not
leaving the docs page — at the cost of an overlay layer and its own client JS, so
it is explicitly optional and deferrable. Whether it ships in v1 is left open (see
the brainstorm's Thread H); the open-full-page button is not optional.

## Decided — metadata lives in a same-name sidecar, not frontmatter

**Decided (sidhantha, 2026-07-07):** an artifact's metadata lives in an optional
**same-name JSON sidecar**, exactly as first-class diagram pages do — never in
frontmatter injected into the `.html` file.

Rationale:

- **HTML files should not need frontmatter surgery.** An artifact is a complete,
  self-contained document — often generated by a tool or an agent, often opened
  and previewed on its own. Prepending YAML frontmatter would corrupt it as a
  standalone HTML file and force every producer to understand our metadata
  convention. A sidecar keeps the artifact pristine.
- **It mirrors the diagram sidecar** (`<XX_name>.meta.json`,
  `diagram-pages.ts:73-79`, `:160-165`), so the loader, the "name suffices —
  sidecar only when needed" authoring guidance, and the user's mental model all
  carry straight over. When the filename alone reads well, no sidecar is required;
  the title derives from the prefix-stripped, title-cased filename
  (`titleFromCleanName`, `:90-92`).
- **The sidecar is also the AI-legibility surface.** Beyond the rendering fields
  (title, description, sidebar label/position, draft), it carries *explicit
  declared values* — purpose, palette, key data — so an agent can understand the
  artifact without parsing its HTML. The concrete shape (a hybrid of typed
  rendering fields plus a convention-driven, loader-opaque `artifact:` block) is
  recommended in the brainstorm's Thread B; the exact reserved-key list inside
  that block is **left open** for the authoring-skill design, its heaviest
  consumer.

Naming is settled at `.meta.json` / `.meta.jsonc` (not a bare `<name>.json`) to
avoid colliding with a colocated data file the artifact itself might fetch. The
implementer must also fix the diagram sidecar's cache bug when cloning:
`.jsonc` sidecars must be added to `dependencyFiles` (the diagram loader omits
this at `diagram-pages.ts:165`), or a `.jsonc` sidecar edit won't invalidate the
section cache.

## Decided — `/artifacts` is a reserved base URL, enforced at config load

**Decided (sidhantha, 2026-07-07):** because `/artifacts` is claimed by the
full-page route, **no docs section may use `artifacts` as its `base_url`**. This
is rejected at **config load time** with an actionable, hard error — and the
limitation is documented in the skills and the user-guide/dev-docs.

**Enforcement point:** inside `loadSiteConfig()`
(`astro-doc-code/src/loaders/config.ts`), at/beside the pages-resolution loop
(`config.ts:209-216`). This runs once, precedes all routing, and is cached
against `site.yaml`. A reserved-base-URL clash is a *config-level* error, so it
follows the established **hard-throw** pattern (the same style as the missing-
theme throw at `config.ts:246-250` and the version gate at `config.ts:172`), not
the `addError`/`addWarning` collection used for content problems — it stops dev,
build, and preview alike.

**Reserved set, not just `artifacts`:** the enforcement covers the full set of
reserved route segments — `artifacts`, `assets`, `content-assets`, `api`,
`editor` — closing the pre-existing hole where nothing today stops a page from
claiming `base_url: "content-assets"` and silently shadow-fighting a serving
route. The recommended mechanism (resurrect and extend the currently-dead
`validateRoutes()` at `config.ts:354-384`, and call it from `loadSiteConfig()`)
is detailed in the brainstorm's Thread G, along with a draft error message.

## Related trust decision (settled here for the record)

**Decided (sidhantha + agent, 2026-07-07):** artifacts are served from the site's
own origin via the `/artifacts/` route and run **unsandboxed as first-party
content** — authored in the repo, reviewed in git, trusted like any layout or
page. `text/html` is scoped to the dedicated route and is deliberately **not**
added to the shared MIME map (`astro-doc-code/src/pages/lib/mime.ts`), so that
colocated `.html` files elsewhere in content (including inside the issue tracker)
do not silently become executable first-party HTML. The authoring skill and docs
must state that an artifact runs with site-origin privileges, so authors never
paste untrusted third-party HTML into a section. Full deliberation: brainstorm
Thread D.

## Decided — the variation-set pattern is skill doctrine

**Decided (sidhantha, 2026-07-08):** the variation-set / options-explorer
pattern — **one artifact carrying 4–10 labeled, genuinely operable design
variations of a single UI element, inside a shared realistic fixture, built to
pick one** — is written into the authoring skill as a first-class pattern
(`references/design-systems.md` section + SKILL.md triage/§0/§1/§3 hooks +
`type: "variation-set"` sidecar keys). Emphasis follows the live-test delta
([../brainstorm/10_variation-comparison/01_delta-analysis.md](../brainstorm/10_variation-comparison/01_delta-analysis.md)):
the shared-fixture realism and the "could be the shipped design" polish bar are
the qualities even a good run misses, and the don't-over-design governor does
not apply to the options themselves. Audit + acceptance trail: subtask 100 and
`agent-log/060`/`070`.

## Decided — the sidecar is design memory; keeping it true is the agent's duty

**Decided (sidhantha, 2026-07-08):** the `artifact:` declaration block stays —
and is promoted, not merely tolerated: its purpose is that the next agent can
extend an artifact or build a design-consistent companion from ~40 lines of
JSON without parsing the HTML. Two duties make it work, both now stated in the
skill: **read the sidecar before touching an existing artifact**, and **update
it in the same change as any HTML edit**. Rationale: this platform is
agents-first — agents perform the edits, humans get observability — so no
human pass will catch a drifted declaration; the drift guard must be the
agent's own discipline (a future `docs-guide` check remains an option if drift
ever bites).

## Decided — self-mode token naming preference

**Decided (sidhantha, 2026-07-08):** a `self`-mode artifact owns its *values*
but should reuse the host theme contract's token *names* for roles the
contract already covers (`--color-bg-primary`, `--color-text-primary`, …)
rather than inventing parallel vocabularies (`--bg`, `--ink`); it is free to
add names for roles the contract lacks (`--color-accent-soft`, `--color-pro`).
A preference, not a hard rule — invented names still render fine — but shared
names keep every artifact's CSS and sidecar legible to the next agent. Origin:
the sidecar-provenance check in
[../agent-log/060_au_variation-capability/03_sidecar-provenance-check.md](../agent-log/060_au_variation-capability/03_sidecar-provenance-check.md).
