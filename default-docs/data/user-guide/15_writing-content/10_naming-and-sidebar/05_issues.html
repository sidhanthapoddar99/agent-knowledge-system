Issues — File Tree → Sidebar

An issue is a folder, and each of its sub-folders has its own naming rule and its own sidebar treatment. Unlike docs — where the order prefix is stripped and the label is the frontmatter title — the issue tracker keeps the number and relocates it to a leading badge. Prefixes are optional here: a loose parse, never a build error. This page maps every sub-folder on disk to exactly what the detail-page sidebar renders.

The whole issue

The detail sidebar opens with fixed “This issue” panels, then one collapsible group per populated sub-folder. Section order is fixed by the layout: Brainstorm · Notes · Subtasks · Agent log · Agent memory.

Leading marks — the legend

Before the walkthrough, here is the full vocabulary of marks that sit at the start of an issue-sidebar row, before the label. These are issue-sidebar concepts — the docs sidebar strips the prefix and shows no leading badge (the folder chevron is the only leading mark it shares). Colour is reserved for status meaning; the neutral NN badge and the chevron/count stay grey. They recur in context throughout the per-sub-folder sections below; this is the one-stop reference. (The trailing type glyphs — the mark for a non-markdown page — are documented in the Suffix Icons artifact.)

Number & iteration badges
05 theme modes explainer

Neutral NN badge issue sidebar

The order prefix, relocated to a leading badge (docs throw the prefix away; issues keep it). Padded to two digits. Appears on normal files, on folders, and on agent-log 0NN_ meta files (goal, summary, task-list…). Always neutral grey — order, not status.

#3not started
#3in progress
#3success
#3failed

#<iteration> badge — status-tinted issue sidebar

An agent-log milestone (a file carrying an iteration field) replaces its NN badge with #N, tinted by the milestone's status: grey = not-started, blue = in-progress, green = success, red = failed. A non-iteration 0NN_ meta file stays on the neutral NN badge — the tint is a milestone signal only.

Agent-log kind symbols

An agent-log activity folder is named NNN_<code>_<name>. When the two-letter code is a recognised kind, its symbol is drawn between the folder's NN badge and its name. Five kinds ship as framework defaults; a project can declare more in an issue's settings.json (agentLogKinds) drawing from a curated symbol palette. Unrecognised code → no symbol, and the name keeps the raw code.

The strict form is NNN_YYYY-MM-DD_author.md (date + author parsed from the name); the looser NNN_<slug>.md is also accepted — the id comes from the leading digits and date/author fall back to frontmatter. The 5-level nesting demo & the depth cap

Folders may nest up to 5 levels deep — a hard cap (MAX_SUBFOLDER_DEPTH = 5); 3 is the recommended ceiling. The folder’s descendant count sits on its row; the numeric prefix badge repeats at each level.

Inside an activity folder — 0NN_ meta vs MNN_ milestone

Edge-case gallery

The corners — what shows, what sorts where, and the one thing that never happens (a hard error).

Prefix present

02reference

02_reference → neutral 02 badge + de-prefixed label. The number is kept, just relocated.

Prefix absent

no prefix scratch

No badge at all, and the row sorts last (+∞). Perfectly valid — issue prefixes are optional.

One digit — not a prefix

5 draft

5_draft has only 1 digit; the grammar needs 2–5. It’s treated as a plain name, no badge, sorts last.

Six digits — not a prefix

123456 draft

123456_draft exceeds 5 digits, so it isn’t an order prefix — the whole thing is the name.

Width-mixed value sort

70test
200stress

70_ before 200_ — ordered by numeric value (70<200), never by string.

Unknown kind code

70nt test

70_nt_test: nt isn’t a declared kind → no symbol, and the label keeps the code.

Meta vs milestone badge

00goal
#1loader reader

0NN_ slot = neutral 00; an iteration: file = status-tinted #1.

Type glyph

03dashboard

A non-markdown note (.html artifact / diagram) gets a trailing glyph. Markdown stays unmarked.

Color tint

01decided architecture

A color: frontmatter value tints just that row’s label — user-defined meaning.

Nesting cap

Folders nest to 5 (hard) / 3 (recommended). A 6th folder level is warned & ignored, not fatal.

Failure mode — there is none

Unlike docs (a missing prefix throws at build), issue sub-docs parse loosely and never throw. Worst case: no badge, sorts last.

Route collision

Two files that reduce to the same slug (03_plan.md + 03_plan.html) are warned — rename one so it doesn’t silently shadow the other.

The docs contrast: in a docs/ section a file with no NN_ prefix is a hard ParserError at build. In the issue tracker the same file is fine — it just renders unnumbered and sorts last. Optional everywhere, fatal nowhere.
