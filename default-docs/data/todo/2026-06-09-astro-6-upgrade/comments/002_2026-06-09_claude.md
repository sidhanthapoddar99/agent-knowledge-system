# Does `./start` reinstall/update dependencies the way it self-updates the framework?

**No.** `./start` self-updates the *framework source code* via git, but it never updates or reinstalls *dependencies* when they change. The two are decoupled, and that's a real gap for an upgrade like this one.

## What `./start` actually does (from the script)

1. **`update_check()` (step 0, lines 29–78)** — `git fetch`, and if the branch is behind upstream and fast-forwardable, prompts `Y/n` to `git pull --ff-only`. This updates **only the tracked source** (including `package.json` / the lockfile *as files*). It does **not** act on what changed.
2. **Install (step 2, lines 109–113)** — guarded by `if [ ! -d node_modules ]`. It runs `bun install` / `npm install` **only when `node_modules` is entirely missing**. If `node_modules` exists, the install step is skipped outright.
3. **Build sanity check (step 3)** then runs against whatever is currently installed.

## The consequence

After a pull that bumps a dependency (exactly this Astro 5→6 upgrade):

- `node_modules` still exists → **install is skipped** → the new `package.json` / lockfile are on disk but **not applied**.
- The build sanity check runs against the **old, stale** dependency tree — so it either passes misleadingly (old Astro still installed) or fails confusingly.
- There is **no** mtime/hash check on `package.json` or the lockfile, **no** `--frozen-lockfile` verification, and **no** post-pull reinstall.

`./start clean` doesn't rescue this either: it wipes `.astro/`, `dist/`, `node_modules/.vite/` (build caches) but **not `node_modules/` itself** (script lines 83–92), so it still won't trigger a reinstall.

## To actually pick up this upgrade

Manual, today:

```bash
cd astro-doc-code
rm -rf node_modules
bun install        # or: npm install
```

(or just `bun install`, which reconciles an existing tree against a changed lockfile).

## Suggested fix (file as a subtask if we act on it)

After a successful `git pull`, detect a changed `package.json` / lockfile (compare pre/post-pull hash) and reinstall — or unconditionally run `bun install` in preflight (near-instant when already satisfied; `bun install --frozen-lockfile` keeps it honest). That would make `./start` update *code and deps* together, matching the mental model the self-update already implies.
