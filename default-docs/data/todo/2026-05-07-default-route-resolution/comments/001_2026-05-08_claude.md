Fixed by making position sort hierarchical in `astro-doc-code/src/loaders/data.ts`.

Root cause: `sortContent('position')` compared only `data.sidebar_position` (a single number derived from the file's XX_ prefix). Files at depth 2 in different folders all have position=1 (e.g. `05_getting-started/01_overview.md`, `35_plugins/01_to_be_written.md`), so the comparator returned 0 and the tiebreaker fell to glob/readdir order — which is filesystem-dependent and explains why `/user-guide` could land on `/user-guide/35_plugins/01_to_be_written`.

Fix: build a per-segment position tuple from `relativePath` (each folder segment contributes its own XX_ prefix, file segment uses `sidebar_position` so frontmatter overrides still win), compare lexicographically. Tie-break on `relativePath` for full determinism. `05_getting-started/01_overview.md` → tuple `[5, 1]` now reliably beats `35_plugins/01_to_be_written.md` → `[35, 1]`.

The default-redirect on line 45 of `pages/[...slug].astro` (`allContent[0]`) is unchanged — it just gets a correctly-ordered array now. Sidebar uses its own hierarchical `buildSidebarTree` and was never affected.

Build green: 413 pages.
