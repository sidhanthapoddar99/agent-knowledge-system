# Storage & Disk Footprint

The framework is designed to be installed **once per project** — so if you run ten documentation projects, you'll have ten clones. This page explains what that actually costs on disk (much less than it looks) and how to keep it small.

## What a clone weighs

| Piece | Approx. size | Notes |
|---|---|---|
| Git history (packed) | ~6 MB | Full history; skippable with a shallow clone |
| Tracked files | ~10 MB | Framework source, bundled docs, plugin |
| `astro-doc-code/dist/` | ~115 MB after a build | Build output — gitignored, never cloned, safe to delete anytime |
| `astro-doc-code/node_modules/` | ~420 MB *apparent* | Installed by `./start` — **mostly shared across projects**, see below |

So the download is roughly **16 MB** (or ~10 MB shallow), and the big numbers you see later in a disk-usage tool are local build artifacts and dependencies — which are cheaper than they appear.

## Shallow clone — the recommended install

As a consumer you don't need the framework's git history. Clone shallow:

```bash
git clone --depth 1 https://github.com/sidhanthapoddar99/agent-knowledge-system.git
```

This stays ~10 MB no matter how long the framework's history grows. The `./start` update check still works — `git pull` deepens the clone as needed when you accept an update.

Use a full clone only if you're contributing to the framework itself (the framework's own issue tracker derives its `updated` dates from git history, so development clones need the history).

## node_modules is shared across projects — if you use Bun

The `~420 MB` under `astro-doc-code/node_modules/` is the one number that looks alarming when multiplied by N projects. With **Bun** (the recommended runner, what `./start` prefers), it isn't multiplied.

### How Bun does it

Bun keeps a single global, content-addressed package cache at `~/.bun/install/cache`. When `bun install` runs in a project, its default backend doesn't *copy* files into `node_modules` — it creates **hardlinks** to the cached files. A hardlink is a second directory entry pointing at the same bytes on disk, so:

- The first project to install a package pays its disk cost once, into the global cache.
- Every subsequent project's `node_modules` references those same bytes. The marginal cost per additional project is directory structure and inodes — a few MB, not 420.
- Disk-usage tools report the full size *per directory* because each hardlink looks like a complete file. Ten projects showing "420 MB each" can genuinely occupy ~420 MB total.

You can verify this yourself — a link count above 1 means the file's bytes are shared:

```bash
stat -c 'links=%h' astro-doc-code/node_modules/astro/package.json
```

Two cases where Bun silently falls back to full copies: the project lives on a **different filesystem/mount** than `~/.bun` (hardlinks can't cross devices), and platforms where hardlinking isn't available. Both are rare; there's nothing to configure either way.

The global cache itself grows over time (it keeps every version ever installed). Prune it with:

```bash
bun pm cache rm
```

### Is this available with plain Node / npm?

Node itself doesn't manage packages — dedup is a property of the *package manager*:

| Package manager | Shared store? | How |
|---|---|---|
| **Bun** | ✅ yes, by default | Global cache + hardlinks (described above) |
| **npm** | ❌ no | Every project gets full copies; the global cache only speeds downloads, not disk |
| **pnpm** | ✅ yes, by design | Content-addressable store at `~/.local/share/pnpm/store` + hardlinks — the same idea as Bun's; it's pnpm's headline feature |
| **Yarn (Berry, PnP mode)** | ✅ different approach | No `node_modules` at all — packages resolve from zip archives in a cache |

So if you can't use Bun, **pnpm** gives you the same cross-project dedup. The `./start` wrapper's fallback is npm (maximum compatibility), which means on an npm-only machine each project really does carry its own full `node_modules` — that's the one setup where the multiplication is real.

## Quick wins if disk is tight

- `rm -rf astro-doc-code/dist/` in projects you're not actively serving — it's regenerated by the next build.
- `bun pm cache rm` to prune old package versions from the global cache.
- Shallow-clone new installs (`--depth 1`).
- Optimize images before committing them (the `agent-ks img` workflow in the [Claude skills](05_claude-skills.md) toolkit; embedding basics in [Asset Embedding](../15_writing-content/03_asset-embedding.md)) — so your *content* doesn't become the heavy part.
