The worktree escape

How agent-ks, run inside an agent's sandbox worktree, ended up writing a file into the real tracker — found twice, independently, during the 6-arm benchmark.

What happened

agent-knowledge-system/                      ← the REAL repo
├── .env                                     ③ …and finds THIS one
├── default-docs/data/todo/                  ④ CONFIG_DIR resolves here → comment lands in the real tracker
└── .claude/worktrees/
    └── agent-<id>/                          ← the agent's isolated copy
        ├── .git                             (a FILE pointing at the outer repo — worktrees don't get their own)
        ├── .env — MISSING (gitignored, so never checked out)  ① CLI needs .env…
        └── default-docs/data/todo/          where the write SHOULD have gone
                                             ② _env.mjs walks UP looking for one…

Why it happens

  1. .env is gitignored, so a fresh worktree never contains one — by design, not by accident.
  2. _env.mjs resolves the content root by walking upward from cwd until it finds a .env. Nothing tells it that .claude/worktrees/agent-…/ is a boundary.
  3. The walk exits the sandbox, hits the outer repo's .env, and every path after that — CONFIG_DIR, the tracker, the write target — belongs to the real checkout.
  4. The CLI then does exactly what it was asked, in exactly the wrong place: agent-ks issue add-comment created the file in the real rebrand issue.

The incident, by the numbers

2 / 3skills arms hit it (sonnet, opus) — independent reproductions
2 / 2caught it themselves (git-status discipline) and reverted
0stray files left in the real repo (verified after each run)
2distinct recoveries: explicit --tracker vs a worktree-local .env

Fix options — either suffices

A · Stop the walk at a worktree boundary

In a worktree, .git is a file, not a directory. When the upward walk passes a directory whose .git is a file (or when git rev-parse --git-dir ≠ --git-common-dir), stop there: the sandbox root is the project root, found .env or not. Fails safe — no .env means the CLI's existing "no .env found" error, which already tells the user what to do.

B · Make the skill mandate it

One line in agent-ks-issues/agent-ks-docs: inside a worktree, always pass --tracker (or write a local .env) before any CLI write. Cheaper, but relies on every future agent reading the rule — the bug stays loaded for anyone who doesn't.

Verdict from the audit: do A in _env.mjs (mechanical, fails safe), and add B's sentence to the skill anyway — defense in depth for the one command class that writes. Full narrative: agent-log/020_au_agent-benchmark/101_run-and-metrics.md.
