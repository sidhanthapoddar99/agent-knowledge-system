# Distribution channels

Companion to `notes/architecture/04_distribution-single-binary.md` (which covers cross-compile mechanics). This note covers the *channels* — how end-users actually get the binary, plus update + version-pinning policies. Lifted from `2026-04-26-framework-as-cli-tool` design and updated for the runtime-migrated reality.

## Channel matrix

| Channel | Audience | One-line install | Update path |
|---|---|---|---|
| `curl \| sh` (Linux/macOS) | Default for Unix | `curl -sSf https://doc-engine.dev/install.sh \| sh` | `doc-engine upgrade` or re-run installer |
| PowerShell (Windows) | Windows users | `iwr -useb https://doc-engine.dev/install.ps1 \| iex` | `doc-engine upgrade` |
| Homebrew (macOS, Linux) | Brew users | `brew install <tap>/doc-engine` | `brew upgrade doc-engine` |
| GitHub releases | Air-gapped, restricted env | Download archive, extract, place on PATH | Manual |
| Docker (registry) | Production, CI | `docker pull doc-engine:latest` | `docker pull` |
| `go install` | Go developers | `go install github.com/<org>/doc-engine@latest` | `go install ...@latest` |
| Direct binary | Embedded in another tool | Vendor the binary in your distribution | Vendor's responsibility |

`goreleaser` produces all of these from one CI run on tag push.

## Install script behaviour (`install.sh`)

Resolves OS + arch, fetches latest version pointer, downloads archive, extracts to `~/.local/bin/` (overrideable via `DOC_ENGINE_INSTALL_DIR`), prints PATH-setup hint. Idempotent — re-running upgrades.

Critical behaviours:

- Verify archive checksum against published `checksums.txt`
- Refuse to install to a system location without `sudo` (no surprise root-level changes)
- Print structured output (so a user piping to `tee` gets a clean log)
- Set `DOC_ENGINE_VERSION` env hint after install for sanity-checks

## PowerShell installer behaviour (`install.ps1`)

- Default install location: `%LOCALAPPDATA%\Programs\doc-engine\`
- Add to user `PATH` automatically (with prompt)
- No admin required for default location
- Same checksum verification

## GitHub releases

Each release ships:
- Per-platform archives (5–8 archives: linux/macos/windows × amd64/arm64)
- `checksums.txt`
- Source tarball
- Release notes auto-generated from commit history (or hand-written)

## Docker registry

Image namespace: `<org>/doc-engine` on Docker Hub + GHCR.

Tags:
- `:latest` — latest stable
- `:0.5.0`, `:0.5`, `:0` — semver tags
- `:edge` — latest commit on main (auto-built per push)
- `:main-<sha>` — pinned commits

Multi-arch manifest: `linux/amd64`, `linux/arm64`. Built via `docker buildx`.

## Versioning policy

| Component | Versioning | Notes |
|---|---|---|
| Binary | SemVer | Major bumps signal breaking changes to `site.yaml` schema or theme contract |
| Theme contract (required_variables) | Lockstep with binary major | Theme contract change is breaking |
| `site.yaml` schema | Lockstep with binary major | Schema break = major bump |
| Tracker schema (settings.json, frontmatter) | Lockstep with binary major | Schema break = major bump |
| Plugin (agent-ks) | Independent SemVer | Decoupled; can ship without binary release |
| Vite frontend bundle | Embedded; tied to binary | No independent versioning |

## Update flow

```
$ doc-engine upgrade
[doc-engine] checking for updates...
[doc-engine] current: 0.5.0
[doc-engine] latest:  0.6.0
[doc-engine] release notes: https://github.com/<org>/doc-engine/releases/v0.6.0
[doc-engine] downloading doc-engine-darwin-arm64 (29 MB)...
[doc-engine] verifying checksum... ok
[doc-engine] replacing /Users/sid/.local/bin/doc-engine
[doc-engine] ✓ upgraded 0.5.0 → 0.6.0
```

Atomic: download to `.new`, verify, rename. Rollback via `doc-engine upgrade --rollback` (keeps last N versions in `~/.cache/doc-engine/binaries/`).

For Homebrew users: `brew upgrade doc-engine` is the canonical path; `doc-engine upgrade` defers to brew if it detects brew installation (refuses to clobber brew's binary).

## Per-project version pinning

Optional in `doc-engine.toml`:

```toml
[engine]
version = "0.5.0"          # this project requires exactly 0.5.0
# or
version_constraint = "^0.5" # this project accepts 0.5.x
```

Behaviour when binary version doesn't match:

- `version` exact match required → error if mismatched, suggest `doc-engine use 0.5.0` (which downloads + caches 0.5.0 alongside current)
- `version_constraint` SemVer range → warning if outside, prompt to download matching

Multiple binaries cached under `~/.cache/doc-engine/binaries/<version>/`. Wrapper at `~/.local/bin/doc-engine` is itself a small script that resolves the project's required version and execs the appropriate cached binary. (Or the binary itself does the dispatch — implementation detail.)

## Air-gapped deployment

For environments without internet:

1. Download binary archive on a connected machine
2. Verify checksum
3. Transfer archive to target (scp, USB, internal repo)
4. Extract; place on PATH
5. Run

No outbound network calls during normal operation. Optionally disable update checks via `doc-engine config set update.check false`.

## Telemetry stance

**No telemetry by default.** No outbound calls beyond explicit `doc-engine upgrade` checks. Errors print to stderr; `doc-engine doctor` produces a local diagnostic bundle the user can manually attach to bug reports.

If telemetry ever ships, it's:
- Opt-in (default off)
- Anonymised (no project paths, no content)
- Self-hostable (point at internal collector via env var)

## Compatibility window

A given binary version supports:
- Its own major's `site.yaml` schema (forward-compatible within the major)
- The previous major's schema, with a deprecation warning at boot
- The next major's schema rejected with a clear "binary too old" error

This three-version window means smooth upgrades for users who skip a release.
