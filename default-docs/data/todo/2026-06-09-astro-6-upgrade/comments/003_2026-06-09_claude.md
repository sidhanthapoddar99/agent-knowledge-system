# Cancelled — not worth it on its own merits

Cancelling after the discovery/documentation pass. The notes (`00-astro-update`, `01-astro-update-impact`, `03-astro-upgrade-benefits`) are **kept** — they're the value of this issue and the decision input for any future revisit.

**Why cancel:**

- **Upside-down cost/benefit.** No perceptible speed win (the big Rolldown build gains are Vite 8 / Astro 7, not this bump) and no new features this framework can use (it rolls its own `marked`+`shiki` content pipeline, so Astro 6's content/collection/adapter additions are inert here).
- **The one real prize is conditional.** Retiring the custom `moduleGraph.invalidateModule` SSR workaround only pays off *if* Astro 6.3.4+'s upstream fix covers our git-ref-driven invalidation path — still ⚠️ unverified (see `notes/01-astro-update-impact/03_loaders-ssr-and-cache.md`).
- **Real risk for little reward.** A major bump dragging in Vite 7 touches our highest-risk code (SSR module-isolation workaround), with a silent-failure mode. We'd spend testing effort mostly to *defend the status quo*.
- **Dominated by a bigger decision.** `2026-05-08-runtime-stack-migration` proposes replacing Astro with a Go + Vite binary. If that's the direction, this bump is throwaway. The real question is "are we staying on Astro at all?" — that should be answered before any version bump.

**Not a permanent "never":** version debt compounds (skipping 6 → 7 → 8 gets painful). If we commit to staying on Astro and happen to be in the build/deps anyway, revisit this — the upgrade-steps and impact notes are ready to execute against.
